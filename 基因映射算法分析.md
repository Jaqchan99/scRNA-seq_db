# 基因映射算法分析报告

## 一、新旧算法对比

### 1.1 旧算法（mapping_service.py 原实现）

**核心特点：**
- **数据源**：依赖外部 API（mygene.info）
- **映射方式**：在线实时查询
- **输入要求**：需要 `gene_symbol` 列
- **处理流程**：
  1. 从 `adata.var['gene_symbol']` 获取基因符号
  2. 逐个调用 mygene.info API 查询
  3. 返回 Ensembl ID
  4. 使用缓存避免重复查询

**关键代码逻辑：**
```python
# 逐个查询每个基因符号
for symbol in gene_symbols:
    result = self._query_gene_mapping(symbol)  # API调用
    # 检查返回的 ensembl_id
```

### 1.2 新算法（jupyter_gene_mapping.txt）

**核心特点：**
- **数据源**：本地预处理文件
  - `id_convert_fp`: 基因名转换字典（包含 @ 分隔的多个 alignment）
  - `gtf_fp`: GTF 文件（包含 gencode.vM32 数据集）
- **映射方式**：本地批量处理
- **输入要求**：直接使用 `adata.var_names`（基因名）
- **处理流程**：
  1. 加载本地映射表（id_convert, gtfM1_M35）
  2. 展开 @ 分隔的多个 alignment
  3. 通过 id_convert 将基因名转换为 alignment
  4. 与 gtfM32 合并获取标准基因名
  5. 处理多 symbol 情况（优先使用原名称，否则使用映射结果）
  6. 更新 adata.var_names

**关键代码逻辑：**
```python
# 1. 展开 @ 分隔的多个 alignment
id_convert_['alignment'] = id_convert_['alignment'].str.split('@')
id_convert_ = id_convert_.explode('alignment')

# 2. 合并转换
var_name_ = pd.merge(var_name_, id_convert_, how='left', on='gene_name')
gene_id_M32 = pd.merge(var_name_, gtfM32, how='left', on='alignment')

# 3. 处理多 symbol 情况
if row['gene_name_x'] in row['gene_name_y']:
    return [row['gene_name_x']]  # 优先使用原名称
```

## 二、为什么旧算法映射率为 0？

### 2.1 可能的原因分析

#### 原因 1：数据列不匹配
- **问题**：旧算法要求 `adata.var['gene_symbol']` 列存在
- **实际情况**：h5ad 文件可能直接使用 `var_names` 作为基因名，没有单独的 `gene_symbol` 列
- **结果**：代码在第 51-58 行直接返回错误，映射率为 0

#### 原因 2：API 查询失败
- **问题**：依赖外部 API，可能因为以下原因失败：
  - 网络连接问题
  - API 限流或超时
  - 物种设置错误（代码中硬编码为 "mouse"）
  - API 返回格式变化
- **结果**：即使有 `gene_symbol` 列，API 查询也可能全部返回 `unmapped`

#### 原因 3：基因名格式不匹配
- **问题**：h5ad 文件中的基因名格式可能与 mygene.info 期望的格式不一致
  - 例如：可能是 Ensembl ID 而非基因符号
  - 或者使用了非标准的命名方式
- **结果**：API 无法识别，返回空结果

#### 原因 4：处理逻辑缺陷
- **问题**：旧算法只处理 `gene_symbol` 列，忽略了 `var_names` 本身
- **实际情况**：很多 h5ad 文件直接将基因名存储在 `var_names` 中
- **结果**：即使文件有基因名，算法也无法使用

### 2.2 新算法为什么能达到 100%？

#### 优势 1：直接使用 var_names
- 新算法直接使用 `adata.var_names`，不依赖特定列名
- 更符合 h5ad 文件的实际结构

#### 优势 2：本地映射表覆盖全面
- 使用预处理的本地映射表，覆盖了常见的基因名变体
- 通过 @ 分隔支持一个基因名对应多个 alignment
- 通过 gtfM32 提供标准化的基因名映射

#### 优势 3：多步骤映射策略
- 第一步：通过 id_convert 转换（支持多个 alignment）
- 第二步：与 gtfM32 合并获取标准基因名
- 第三步：智能处理多 symbol 情况（优先保留原名称）

#### 优势 4：容错机制
- 如果映射失败，使用原基因名而非 UNKNOWN（在某些情况下）
- 支持部分匹配和模糊匹配

## 三、Cell Type Mapping 算法问题分析

### 3.1 当前实现的问题

#### 问题 1：依赖外部 API（与旧基因映射相同）
- **现状**：使用 Zooma API 进行在线查询
- **风险**：
  - 网络依赖性强
  - API 可能不稳定或限流
  - 查询速度慢（每个细胞类型需要单独请求）

#### 问题 2：数据列要求严格
- **现状**：要求 `raw_cell_type_label` 列存在
- **问题**：如果 h5ad 文件使用其他列名（如 `cell_type`, `celltype`, `annotation`），算法无法识别

#### 问题 3：映射策略单一
- **现状**：只使用 Zooma API，没有备选方案
- **问题**：
  - 如果 Zooma 无法识别某个细胞类型，直接返回失败
  - 没有本地映射表作为备选

#### 问题 4：缺乏预处理和标准化
- **现状**：直接查询原始细胞类型名称
- **问题**：
  - 细胞类型名称可能有多种变体（大小写、空格、缩写等）
  - 没有预处理步骤统一格式
  - 例如："T cell" vs "T-cell" vs "Tcell" 可能被识别为不同的类型

#### 问题 5：置信度处理不足
- **现状**：只检查是否有 CL ID，不评估置信度
- **问题**：
  - 低置信度的映射也被接受
  - 没有过滤机制排除不可靠的映射

### 3.2 改进方向建议

#### 改进方向 1：构建本地细胞类型映射表（最重要）
**方案**：
- 创建类似基因映射的本地映射表
- 包含常见细胞类型名称变体到 CL ID 的映射
- 支持模糊匹配和同义词

**实现思路**：
```python
# 类似基因映射的结构
cell_type_mapping = {
    'T cell': 'CL:0000084',
    'T-cell': 'CL:0000084',
    'Tcell': 'CL:0000084',
    'T lymphocyte': 'CL:0000084',
    # ... 更多映射
}
```

#### 改进方向 2：多数据源融合
**方案**：
- 优先使用本地映射表
- 本地映射失败时，再查询 Zooma API
- 可以集成其他数据源（如 Cell Ontology 直接查询）

**优先级**：
1. 本地映射表（最快、最可靠）
2. Zooma API（作为备选）
3. Cell Ontology 直接查询（最后备选）

#### 改进方向 3：名称预处理和标准化
**方案**：
- 统一大小写
- 去除多余空格
- 处理常见缩写和同义词
- 例如："CD4+ T cell" -> "T cell" -> 查找映射

**实现示例**：
```python
def normalize_cell_type_name(name):
    # 转小写
    name = name.lower().strip()
    # 统一分隔符
    name = name.replace('-', ' ').replace('_', ' ')
    # 处理常见缩写
    name = name.replace('cd4+', 'cd4 positive')
    # ...
    return name
```

#### 改进方向 4：支持多种列名
**方案**：
- 自动检测可能的细胞类型列名
- 支持：`raw_cell_type_label`, `cell_type`, `celltype`, `annotation`, `cell_annotation` 等

**实现示例**：
```python
def find_cell_type_column(adata):
    possible_columns = [
        'raw_cell_type_label', 'cell_type', 'celltype',
        'annotation', 'cell_annotation', 'cell.type'
    ]
    for col in possible_columns:
        if col in adata.obs.columns:
            return col
    return None
```

#### 改进方向 5：置信度评估和过滤
**方案**：
- 评估映射的置信度
- 设置置信度阈值（如 medium 以上）
- 低置信度映射标记为 ambiguous

**实现思路**：
```python
if confidence in ['high', 'good']:
    status = "mapped"
elif confidence == "medium":
    status = "mapped"  # 或 "ambiguous" 根据需求
else:
    status = "unmapped"
```

#### 改进方向 6：批量处理和缓存优化
**方案**：
- 批量查询相似细胞类型
- 增强缓存机制（持久化到文件）
- 避免重复查询相同细胞类型

#### 改进方向 7：模糊匹配算法
**方案**：
- 使用字符串相似度算法（如 Levenshtein 距离）
- 当精确匹配失败时，尝试模糊匹配
- 设置相似度阈值

**实现示例**：
```python
from difflib import SequenceMatcher

def fuzzy_match(cell_type, mapping_table, threshold=0.8):
    best_match = None
    best_score = 0
    for key, cl_id in mapping_table.items():
        score = SequenceMatcher(None, cell_type.lower(), key.lower()).ratio()
        if score > best_score:
            best_score = score
            best_match = (key, cl_id)
    if best_score >= threshold:
        return best_match
    return None
```

## 四、总结

### 4.1 基因映射算法改进总结

**旧算法失败的根本原因**：
1. 数据列依赖问题（要求 `gene_symbol` 列）
2. 外部 API 依赖性强
3. 基因名格式不匹配

**新算法成功的关键**：
1. 直接使用 `var_names`，不依赖特定列
2. 本地映射表，覆盖全面
3. 多步骤映射策略，容错性强

### 4.2 Cell Type Mapping 改进优先级

**高优先级（必须改进）**：
1. ✅ 构建本地细胞类型映射表（类似基因映射）
2. ✅ 支持多种列名自动检测
3. ✅ 名称预处理和标准化

**中优先级（建议改进）**：
4. 多数据源融合（本地 + API）
5. 置信度评估和过滤
6. 批量处理和缓存优化

**低优先级（可选改进）**：
7. 模糊匹配算法
8. 机器学习辅助映射

### 4.3 实施建议

1. **短期**：参考基因映射的新算法，构建本地细胞类型映射表
2. **中期**：实现多数据源融合和名称预处理
3. **长期**：优化算法性能，添加机器学习辅助
